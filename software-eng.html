Software Life Cycle: Development Phase:

    Requirements analysis:

    • Consists of compiling and analyzing the needs of the software user; negotiating with the project's stakeholders over trade-offs between wants, needs, costs, and feasibility; and finally, developing a set of requirements that identify the features and services that the finished software system must have. The goal of which is to specify what services the proposed system will provide, to identify any conditions (time constraints, security, and so on) on those services, and to define how the outside world will interact with the system.

    Design:

    • Involves creating a plan for the construction of the proposed system. The result of the design phase is a detailed description of the software system's structure that can be converted into programs.

    Implementation:

    • Involves the actual writing of programs, creation of data files, and development of databases.

    Testing:

    • Essentially equated with the process of debugging programs and confirming that the final software product was compatible with the software requirements specification.

Modular Implementation:

  • Is based on the goal during the initial software desgin; either as imperative paradigms, in which modules appear as functions, or in object-oriented paradigms, which uses objects as the basic modular constituents.

Unifed Modeling Language (UML):

  • Modern collection of tools that have been developed with the object-oriented paradigm in mind.

Use Case:

  • The interactions between the system and its user.

Use Case Diagram:

  • Depicts the proposed system as a large rectangle in which interactions between the system and its users are represented as ovals and users of the systems are represented as stick figures.

Data Relationships:

    One-to-One:

    • Indicates the association multiplicities occurs as only one to one.

    One-to-Many:

    • Indicates the association multiplicities occurs as one to many.

    Many-to-Many:

    • Indicates the association multiplicities occurs as many to many.

Design Pattern:

  • A predeveloped model for solving a recurring problem in software design.

Software Quality Assurance (SQA):

  • Groups that are charged with overseeing and enforcing the quality control systems adopted by the organization.

Software Testing:

    Glass Box Testing:

    • The software tester is aware of the interior structure of the software and uses this knowledge when designing the test.

    Black Box Testing:

    • Refers to tests that do not rely on knowledge of the software's interior composition.

    Beta Testing:

    • Falls within the black-box category; where a preliminary version of the software is given to a segment of the intended audience with the goal of learning how the software performs in real-life situations before the final version of the product is solidified and released to the market.

    Alpha Testing:

    • Similar testing to beta testing but is performed at the developer's site.

Documentation:

    User Documentation:

    • The purpose is to explain the features of the software and describe how to use them.

    System Documentation:

    • The purpose is to describe the software's internal composition so that the software can be maintained late in its life cycle.

    Technical Documentation:

    • The purpose is to describe how a software system should be installed and serviced.

Intellectual Property Law:

    Software License:

    • Where the rights of the developer are formally expressed in legal terms.

    Software Disclaimer:

    • Statements in the software licenses that state the limitations of a developer's liability.

    Best defense against software liability claims:

    • To apply sound software engineering principles during the software's development, to use a level of care compatible with the software's application, and to produce and maintain records that validate these endeavors.
